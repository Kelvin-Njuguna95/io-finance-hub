# Impeccable — IO Finance Hub Design Context

This file is the design-system spec for the Finance Hub. Tokens in `src/app/globals.css` must match this spec. If they drift, the spec wins — update `globals.css` to match, not the other way.

---

## Design Context

### Users
- CFO, Accountant, Team Lead, Project Manager at Impact Outsourcing (Kenya-based BPO)
- Internal tool, daily active use during working hours
- Extended sessions — eye fatigue matters; users read and enter data for multiple continuous hours
- Regional: Kenya. Currency: KES + USD. Statutory register: KRA, NSSF, NHIF
- Device: desktop browsers, Chromium-family

### Brand Personality
Three words: **disciplined · legible · unshowy**.

Tone: calm, authoritative, data-dense. Explicitly NOT slick or marketing-flavoured. Boring-but-excellent is the goal. Finance users reward clarity and instant legibility over polish or delight.

### Aesthetic Direction
- Register: treasury / regulatory document, not consumer-SaaS or marketing site
- Reference: Stripe Dashboard, Mercury, Xero (mainstream competent treasury)
- Anti-reference: Linear marketing site, Notion landing page, any "2024 SaaS" dashboard with glossy surfaces, gradient hero sections, or decorative motion
- Surface: light only (see Theme)
- Spatial language: 4pt grid, breathing-stop defaults

### Design Principles
1. **One meaning per token suffix.** If `-muted` means "neutral surface for grey text" at one place, it cannot mean "faded anchor" somewhere else.
2. **Discipline scales better than intensity.** Fewer weights, fewer radii, fewer shadow stops — the system gets stronger the more choice points we remove.
3. **Italics is a signal, not emphasis.** Exactly one italic role in the interface (estimated values). Everything else is upright.
4. **Eye fatigue is a design constraint.** Body ≥15px, contrast ≥4.5:1 body / ≥3:1 UI, generous row heights, tabular-nums on every column users scan.
5. **Brand colour stays out of data.** Navy is the brand; charts use their own palette. Users must never accidentally read brand as a signal.

## Accessibility

- Target: WCAG 2.1 AA
- Contrast: ≥4.5:1 for body text, ≥3:1 for large text and UI components. Enforced structurally through colour tokens
- Structural a11y (not font-driven): body minimum 15px (16 preferred), clear type scale (≥1.25 ratio between levels), `tabular-nums` on currency and IDs, generous table row height
- Focus ring: see Focus Rings section
- Motion: see Motion Policy

## Typography

### Stack
- **Body:** Commissioner 400 (via `next/font/google`, CSS var `--font-commissioner`)
- **Headings:** Commissioner 600
- **Reserved weight:** Commissioner 500 — active nav item and selected tab only. Not introduced without explicit approval.
- **Numeric / mono:** JetBrains Mono 400

### Weights loaded
Commissioner: 400, 600 (plus 500 if/when approved). JetBrains Mono: 400 only. Do **not** preload the full variable range.

### Mono scope
JetBrains Mono applies to:
- All currency amounts (KES, USD)
- Invoice numbers and reference codes
- UUIDs
- Table columns where users read numbers as a vertical stack

Default to Commissioner for everything else. If a value is ambiguously "numeric enough", default to Commissioner and ask.

### Italics policy
Italics is exclusive to **estimated values** (the `revenueEstimated` flag in the revenue model). One italic role, one meaning.

**Strict negative rules (never add without explicit approval):**
- No italics for emphasis
- No italics for foreign words
- No italics for proper nouns
- No italics for report titles
- No italics anywhere else in the UI

The `≈` prefix in Profitability cards is a **secondary accessibility affordance** (screen-reader-announceable, redundancy-by-design). Never remove the `≈` prefix from any component that currently renders it.

If a future design decision tempts adding a second italic role, **stop and ask** — a second role erodes the signal.

### Mono-italic test step
When the font lands, render JetBrains Mono italic at:
- Revenue table row
- Profitability card totals
- Budget line items

If the italic glyphs look mechanically wrong at those sizes, do not ship silently. Propose a fallback: italicise the row context/label and leave the number upright, **or** a muted cell background instead of italic type.

### Type scale
Minimum 1.25 ratio between levels. Specific scale pinned during component polish; the foundation PR only enforces the ratio.

## Theme

**Light only. Dark mode has been explicitly deprecated and removed. A future revisit would be a multi-quarter design project, not a token swap.**

Consequences:
- `.dark {}` block deleted from `globals.css` (not commented out)
- Theme toggle component removed from the UI
- `ThemeProvider` and `next-themes` usage removed from the layout tree (the package is left in `package.json` for a follow-up dep cleanup; no runtime import remains)
- `suppressHydrationWarning` removed from `<html>` — it existed solely for `next-themes`
- Any `useTheme()` consumer updated to drop the check

## Colour Syntax

**`oklch()` everywhere. Hex is banned from shipped CSS.**

### Precision
All oklch values use **3-decimal lightness, 3-decimal chroma, integer hue**:

```css
oklch(0.970 0.005 254)   /* correct */
oklch(0.97 0.005 254)    /* wrong — lightness must be 3 decimals */
oklch(0.970 0.005 254.3) /* wrong — hue must be an integer */
```

### Hex as comment-only
If a token derives from an external brand hex, the hex lives in a comment above the declaration:

```css
/* #1a4d8c */
--primary: oklch(0.295 0.065 254);
```

`oklch` is the source of truth. Never ship a hex value in CSS.

### Browser support
oklch is supported in Chrome 111+, Safari 15.4+, Firefox 113+, Edge 111+. Finance Hub users are Kenya-based internal staff on modern Chromium-family browsers. No fallback layer needed.

## Colour System

### Neutral ramp (10 stops)
Perceptually even lightness stepping, tinted toward the brand hue 254 at chroma 0.005:

```css
--neutral-50:  oklch(0.985 0.005 254);
--neutral-100: oklch(0.955 0.005 254);
--neutral-200: oklch(0.915 0.005 254);
--neutral-300: oklch(0.865 0.005 254);
--neutral-400: oklch(0.805 0.005 254);
--neutral-500: oklch(0.720 0.005 254);
--neutral-600: oklch(0.580 0.005 254);
--neutral-700: oklch(0.430 0.005 254);
--neutral-800: oklch(0.290 0.005 254);
--neutral-900: oklch(0.170 0.005 254);
```

Adjust a specific stop only if it fails WCAG contrast against an adjacent one.

### Brand anchor
**Single anchor:** deep navy.

```css
--primary: oklch(0.295 0.065 254);
```

### Non-negotiable brand rules
- The Finance Hub has **exactly one** brand accent: deep navy. No second brand accent, ever.
- **Gold and electric blue are chart-only colours.** Any UI occurrence outside `/components/charts/*` is a bug.
- Gold and electric blue must not appear as button backgrounds, focus rings, selected states, or brand surfaces.

### Derived anchor recipes
All semantic anchors (`primary`, `success`, `warning`, `danger`, `info`) derive `-soft`, `-soft-foreground`, `-foreground`, `-hover`, `-active` using this exact recipe book:

| Variant | Recipe |
|---|---|
| `-soft` | `color-mix(in oklab, var(--anchor) 12%, white)` |
| `-soft-foreground` | Dark stop from neutral ramp, verified ≥4.5:1 against `-soft` |
| `-foreground` | White or near-white neutral, verified ≥4.5:1 against anchor |
| `-hover` | `color-mix(in oklab, var(--anchor) 93%, white)` |
| `-active` | `color-mix(in oklab, var(--anchor) 93%, black)` |

**Hover exception:** if a `-hover` element sits on a `-soft` background and reads as invisible, lift the hover mix to `88%` for **that specific element only** (via a scoped class or inline var). Do not change the base rule.

### Muted (standalone, not anchor-derived)
`--muted` is a standalone neutral surface; `--muted-foreground` is standalone neutral text. Neither derives from `--primary` or any anchor. This prevents the `primary-muted` vs `muted` suffix collision.

### Suffix discipline (strictly enforced)
One meaning per suffix:
- `-soft` = tinted pale surface derived from an anchor (12% mix)
- `-foreground` = accessible text colour on the anchor (solid, not mixed)
- `-hover` / `-active` = interactive state shifts (93/7 mix)
- `-muted` = standalone neutral surface, never derived from an anchor

If a component needs a colour value that doesn't fit one of these suffixes, stop and ask — don't invent a new suffix.

## Chart Palette (policy only — foundation PR does not implement)

Captured now so future polish PRs don't drift.

### Principles
- **Categorical palette:** a research-backed 6–8 colour set (Observable Plot default / ColorBrewer Set2 / Tableau 10), picked for colourblind safety and optical balance. Not derived from the brand anchor. Electric blue and gold are members of this set but not special.
- **Sequential palettes** (aging buckets, heatmaps): oklch ramps from neutral to a single hue with even lightness stepping.
- **Diverging palettes** (budget variance, delta visualisations): two perceptually opposite hues balanced through neutral (e.g. teal / red).
- **Deep navy (`--primary`) does not appear in any chart.** Charts use their own palette so users never accidentally read brand as a chart signal.

### Scope boundary
Chart palette lives in `/components/charts/*`. No chart colour escapes into brand surfaces; no brand colour escapes into charts.

## Radius / Shape

**Exactly four tokens. No intermediate stops.**

```css
--radius-sm:   0.25rem;  /*  4px */
--radius:      0.375rem; /*  6px — base */
--radius-lg:   0.625rem; /* 10px */
--radius-full: 9999px;
```

### Usage
| Token | Used on |
|---|---|
| `--radius-sm` | Badges, switches, checkboxes, toggle indicators |
| `--radius` (base) | Inputs, buttons, small cards, most UI |
| `--radius-lg` | Dialogs, popovers, larger cards, dropdowns |
| `--radius-full` | Pills only (tabs selected state, chip-style filters) |

### Negative rules
- **No `--radius-md`, `--radius-xl`, `--radius-2xl`.** The differences between adjacent fine-grained stops (4→5→6) are not distinctions the human eye registers on a real screen.
- **No arbitrary `rounded-[Xpx]` Tailwind utilities** in component files. If a primitive needs a non-standard radius, the system is wrong — stop and ask.
- New radius tokens must not be added without explicit approval.

## Elevation / Shadow

**Single overlay elevation token. Everything else is flat.**

```css
--shadow-overlay: /* subtle shadow — see globals.css for exact value */;
```

### Usage
`--shadow-overlay` applies **only** to:
- Popovers
- Dialogs
- Dropdowns
- Tooltips
- Toasts

Cards, buttons, inputs, sidebars, table rows: **flat**. Border-only. No shadow.

### Negative rules
- **Hover states never use shadows.** Hover on cards, buttons, and interactive containers uses background tint changes (the `-hover` recipe), never shadow lift. Card hover-shadow is a consumer-SaaS pattern that contradicts our sparse-motion policy.
- No multi-stop elevation system. `--shadow-elev-1/-2/-3/-hero` tokens are deleted.

## Density / Spacing

**4pt base grid. Tailwind's native scale. Breathing-stop defaults.**

### Defaults
Default reach is **8 / 12 / 16 / 24** (breathing stops). Reach for 4-step only when density genuinely requires it.

### Primitive targets
| Primitive | Sizing |
|---|---|
| Button (default) | `h-9 px-4 py-2` |
| Input / Select / Textarea | `h-9` (unified across form primitives) |
| Table row cell | `py-3 px-4` (12 / 16) |
| Card | `p-6` (24) — breathing room |
| Sidebar nav row | ~40px tall |
| Badge | `px-2 py-0.5` (8 / 2) |

### Explicit off-grid exception
Badge internal padding uses `py-0.5` (2px) because 4pt padding on sub-20px elements renders as over-generous. **This is the only permitted off-grid value anywhere.** If a component needs a non-grid value, stop and ask — the grid is the discipline.

## Focus Rings

**One focus-ring treatment used by every interactive primitive.**

### Token
```css
--ring: oklch(0.627 0.179 254);
```

Lighter saturated navy — visible against light surfaces, matches brand hue without competing with solid primary surfaces.

### Style
```css
:focus-visible {
  outline: 2px solid var(--ring);
  outline-offset: 2px;
}
```

### Rules
- **`focus-visible` only.** Mouse click does not show the ring. Tab / Shift-Tab does.
- **No primitive may opt out.** Every interactive element must show the visible focus-visible outline.
- **Minimum 3:1 contrast** against every adjacent background in the app. Verify during primitive audit; adjust chroma/lightness if any surface fails.
- If a primitive previously used a different focus pattern (`ring-*`, `focus-visible:ring-*`, custom border-colour shifts), migrate to the unified outline treatment.

## Motion Policy

**Sparse. Motion is permitted only on:**
- Status transitions
- Toast entries / exits
- Tab switches
- KPI count-ups

**No decorative motion anywhere.** No hover-lift, no fade-in on mount, no scroll-triggered animations, no skeleton shimmer on interactive elements unless functionally required, no auto-playing carousels.

`@media (prefers-reduced-motion: reduce)`: all animation durations collapse to near-zero.
